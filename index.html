01

Purpose

Script 3 is the core calculation engine of the GHG BEX platform. It takes raw event activity data and matches each row against the best available emission factor from dmhistory, then computes CO₂ equivalent emissions.

Two event streams are supported, controlled by a single flag in Cell 2:

ENV = "bex"  # BernExpo events → reads fcinbex, writes fcoutbex
ENV = "se"   # Sustainable Events (MuuvFestival, XaverAward…) → reads fcinse, writes fcoutse

High-level flow

fcinbex / fcinse
→
Exclusions
→
Normalize keys
→
Tiered match
→
Calculate CO₂
→
fcoutbex / fcoutse

Key design principle: Every input row receives exactly one output row. Rows with no DM match (NoMatch / orphans) still appear in the output with null CO₂, so they are visible in QA for manual follow-up.

Example: petite arvine wine row has DMStatus = "history" → excluded. The DMHistory1 source in the Power Query combine always tags old snapshot rows as "History" — this is how the system keeps audit history without polluting the active matching pool.

05

Key normalization

Before joining, all key columns are normalized on both tables so minor formatting differences don't cause NoMatch. Cell 5

# Applied to: ActivitySource, ActivityEN, ActivityID, Context
1. lowercase("Security / Guarding") → "security / guarding"
2. trim("  articles  ")            → "articles"
3. non-breaking space → regular space
4. collapse multiple spaces        → "chair: without armrest black"  # double → single
5. ContextDate cast to integer      → 2025

The double-space bug: If DMInput has "chair: without armrest black" (2 spaces) AND "chair: without armrest black" (1 space), normalization makes them identical → C3 duplicate. Clean source data in SharePoint to remove extra spaces.

Shadow keys (_k columns)

Normalized versions stored in _k columns used only for the join. Originals are preserved and appear in the output. The _k columns are stripped before writing to fcout.

ActivitySource  →  ActivitySource_k
ActivityID      →  ActivityID_k
ActivityEN      →  ActivityEN_k   # these are never in the output
06

Tiered matching

For each FCIn row, the script finds the single best DM row using a 4-tier priority system. The foundation is always the 3 join keys — tiers only apply once the keys already match.

The 3 join keys (mandatory for any match)

All three must match simultaneously. There is no way to match on ActivitySource alone — the full combination is required to identify a unique activity.

join_keys = ["ActivitySource", "ActivityID", "ActivityEN"]

# When ActivityID is filled:
fcinse:     articles + 150 + "chair: without armrest black"
dmhistory:  articles + 150 + "chair: without armrest black"  → MATCH ✓

# When ActivityID is empty → ActivityEN is the sole identifier:
fcinse:     articles + "" + "security / guarding"
dmhistory:  articles + "" + "security / guarding"  → MATCH ✓
dmhistory:  articles + "" + "security"             → NO MATCH ✗  # not exact

Tier priority — once keys match

T1
Perfect match
Same keys + same Context + same ContextDate. Best possible.
T2
Year match
Same keys + same ContextDate, any Context.
T3
Context match
Same keys + same Context, any year. Tiebreak: most recent ContextDate_dm.
T4
Fallback
Same keys only. Any context, any year. Also used for location rows (no EF needed).

Tiebreak rules — when multiple rows share the same tier

PriorityApplies toRule
1stTier 3 onlyMost recent ContextDate_dm descending
2ndAll tiersMost recent DMImportDate descending
3rdAll tiersSHA256 hash (deterministic, always picks same row)

07

CO₂ calculation

Step 1 — QtyTot

QtyTot = Qty × AssumptionValue_dm

AssumptionValue must never be null. If it is null, QtyTot = null and EmissionsKgCO2eq = null. Always set at minimum AssumptionValue = 1 if no real conversion is needed.

Step 2 — EmissionsKgCO2eq

EmissionsKgCO2eq = QtyTot × EmissionFactor_dm

Real examples

Example A — entrecôte (food, kg-based)
Qty (fcinse)125 portions
AssumptionValue0.2 (kg per portion)
QtyTot125 × 0.2 = 25 kg
EmissionFactor29.1043 kgCO₂e/kg
EmissionsKgCO2eq25 × 29.1043 = 727.6 kgCO₂e
Example B — security / guarding (articles, unitday-based)
Qty (fcinse)2 unitday
AssumptionValue1 (1 unitday = 1 day)
QtyTot2 × 1 = 2 days
EmissionFactor6 kgCO₂e/day (oth25-12032)
EmissionsKgCO2eq2 × 6 = 12 kgCO₂e
Example C — external security (transaction, CHF-based)
Qty (fcinse)500 CHF (rate: 500 CHF/day)
AssumptionValue0.002 (1/500 = 0.002 days per CHF)
QtyTot500 × 0.002 = 1 day
EmissionFactor6 kgCO₂e/day (oth25-12032)
EmissionsKgCO2eq1 × 6 = 6 kgCO₂e

Step 3 — QtyTotUnit (denominator cancellation)

QtyUnitAssumptionUnitResultRule
(empty)kgkgNo QtyUnit → use AssumptionUnit
unitdayday/unitdaydayunitday cancels → keeps "day"
CHFday/CHFdayCHF cancels → keeps numerator "day"
pkm(empty)pkmNo AssumptionUnit → keep QtyUnit
kwhkwhkwhSame on both sides → keep once
chfkwh/chfkwhchf cancels → keeps kwh

How DMHistory is built

DMHistory is rebuilt by a Power Query workflow before each script run, combining 4 sources:

Table.Combine({DMFCSeOut, DMFCBexOut, DMInput, DMHistory1})
SourceOriginDMStatusPurpose
DMInputSharePoint (manual)new / activeHuman-curated factors. The ground truth. You add rows here for new events.
DMFCSeOutPrevious fcoutse (lakehouse)ActivePast SE outputs fed back. Deduped. Enables Tier 1 for recurring events.
DMFCBexOutPrevious fcoutbex (lakehouse)ActiveSame for BEX stream.
DMHistory1Previous dmhistory snapshotHistoryOld snapshot for audit. Tagged History → excluded from matching.

Why DMHistory1 is tagged "History": DMFCSeOut and DMFCBexOut already have the active validated versions. The old snapshot is kept for traceability but must not compete with current active rows in the tier system.

Self-learning loop

The system improves automatically. Each run's output feeds back into DMHistory, enriching future runs.

New event data loaded into fcinse
↓
Script 3 matches + calculates CO₂
First run for new event: mostly Tier 3/4 (borrowing from other contexts)
↓
fcoutse written to lakehouse
↓
Power Query rebuilds DMHistory
fcoutse → DMFCSeOut (Active) + old DMHistory → DMHistory1 (History)
↓
Next run of same event → Tier 1 matches available ★
Context + ContextDate now exist in DMHistory → perfect match

Assumption guide — how to fill DMInput

NEW

The assumption is the bridge between what your event data gives you (QtyUnit) and what the emission factor expects (its unit). Three questions to ask for every row:

Question 1: What does my fcinse give me?
  → Look at Qty and QtyUnit

Question 2: What does the emission factor expect?
  → Look at EmFactUnit from the EMFAID

Question 3: How do I convert one to the other?
  → That conversion = AssumptionValue + AssumptionUnit

The golden rule — denominator cancellation

QtyUnit × AssumptionUnit = EF unit (denominator of AssumptionUnit must match QtyUnit) 
unitday × day/unitday = day  ✅  (unitday cancels)
CHF     × day/CHF     = day  ✅  (CHF cancels)
portion × kg          = kg   ✅  (no denominator needed when QtyUnit is empty)
pkm     × (nothing)   = pkm  ✅  (already in EF unit)

The 3 fields explained

FieldWhat it meansWhen to change it
AssumptionValueThe numeric multiplier. How many physical units per event unit.Always fill — minimum 1 if no real conversion needed. Null = no CO₂.
AssumptionUnitThe unit of the conversion. Denominator must cancel QtyUnit.Set to "EFunit/QtyUnit". Leave empty if QtyUnit already equals EF unit.
AssumptionNoteFree text explaining the logic. Not used in calculation.Always fill — helps future team members understand the assumption.

Car travel / mobility
Qty=37050 pkm → EF per pkm
AssumptionValue = 1
AssumptionUnit = (empty)

Electricity / energy
Qty=21086 kwh → EF per kwh
AssumptionValue = 1
AssumptionUnit = (empty)
Note = "kwh already in EF unit"
Fuel / transaction (CHF→kwh)
Qty=1200 CHF → EF per kwh (rate 8.33 kwh/CHF)
AssumptionValue = 8.333
AssumptionUnit = kwh/chf
Note = "2024 value, rate 8.33 kwh/CHF"

Decision tree — what AssumptionValue to use

Is QtyUnit already the same as the EF unit?
  → YES: AssumptionValue = 1, AssumptionUnit = empty or same unit
  → NO:
      Is it a rate conversion (price per physical unit)?
        → YES: AssumptionValue = 1 / rate  (e.g. 1/500 = 0.002 for 500 CHF/day)
               AssumptionUnit = "EFunit/QtyUnit"  (e.g. day/chf)
      Is it a physical quantity per event unit?
        → YES: AssumptionValue = quantity  (e.g. 0.2 kg per portion)
               AssumptionUnit = "EFunit"  (e.g. kg)

Tier distribution — what good looks like

NEW

Section D of the monitoring output shows how many rows matched at each tier. This is the most important quality signal.

Tier 1 (Context+Year)703 rows (71%) — perfect matches ✅
Tier 2 (Year only)0 rows (0%)
Tier 3 (Context only)0 rows (0%)

What you seeWhat it meansAction
Tier 1 = 100% of non-location rowsPerfect. All activity rows have a DM entry for this exact event + year.None needed ✅
High Tier 2 or Tier 3DM entries exist but for different context or year. Matches are borrowed.Add Context+ContextDate to DMInput for this event
